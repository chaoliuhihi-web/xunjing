# 星河寻境官网运营配置说明

更新时间：2026-06-21

## 目标

官网第一阶段已经可以作为静态站点或 Docker Compose 站点部署。为了支持真实对外运营，预约演示表单支持运行时配置线索 webhook，不需要重新打包前端；Docker Compose 部署默认提供同域 `/api/leads` 本地线索采集服务。

## 线索 webhook

部署产物里包含：

```text
runtime-config.js
```

默认内容：

```js
window.XINGHE_SITE_CONFIG = {
  leadWebhookUrl: ''
};
```

Docker Compose 默认会把 `leadWebhookUrl` 渲染为同域路径：

```js
window.XINGHE_SITE_CONFIG = {
  leadWebhookUrl: '/api/leads'
};
```

上线时把 `leadWebhookUrl` 改为 CRM、企业微信机器人、飞书机器人、后端 API 或自动化平台的 HTTPS 地址：

```js
window.XINGHE_SITE_CONFIG = {
  leadWebhookUrl: 'https://your-crm.example.com/api/xinghe/leads'
};
```

表单会向该地址发送 `POST application/json`，字段包括：

- `name`
- `phone`
- `company`
- `type`
- `message`
- `source`
- `page`
- `url`
- `userAgent`
- `submittedAt`
- `status`

如果未配置 webhook 且没有使用 Compose 默认 `/api/leads`，线索会写入浏览器 `localStorage` 的 `xinghe_xunjing_leads`，状态为 `pending-sync`。这只能作为兜底，不适合作为长期运营方案。

## Nginx 缓存要求

`runtime-config.js` 必须禁止缓存，避免运营人员改 webhook 后浏览器继续使用旧配置：

```nginx
add_header Content-Security-Policy "default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'self'; img-src 'self' data: https:; script-src 'self'; style-src 'self'; connect-src 'self' https:" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "SAMEORIGIN" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;

location = /runtime-config.js {
    add_header Content-Security-Policy "default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'self'; img-src 'self' data: https:; script-src 'self'; style-src 'self'; connect-src 'self' https:" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;
    add_header Cache-Control "no-store";
    try_files $uri =404;
}
```

静态资源可以长期缓存：

```nginx
location ^~ /assets/ {
    add_header Content-Security-Policy "default-src 'self'; base-uri 'self'; object-src 'none'; frame-ancestors 'self'; img-src 'self' data: https:; script-src 'self'; style-src 'self'; connect-src 'self' https:" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=(), payment=()" always;
    add_header Cache-Control "public, max-age=2592000, immutable" always;
    try_files $uri =404;
}
```

注意：Nginx 的 `add_header` 在子 `location` 中不会自动继承上层同类配置，因此配置 `runtime-config.js` 和 `/assets/` 缓存头时，也要重复保留安全响应头。

## Docker 部署

本仓库包含 `Dockerfile`、`ops/nginx.conf` 和 `ops/compose.prod.yml`。

单容器运行：

```bash
docker build -t xinghexunjing-web:latest .
docker run --rm -p 8080:80 \
  -e XINGHE_LEAD_WEBHOOK_URL="https://your-crm.example.com/api/xinghe/leads" \
  xinghexunjing-web:latest
```

生产服务器建议使用 Compose，带自动重启、`/healthz.json` 网站健康检查、`/api/leads` 线索采集和持久化卷：

```bash
export XINGHE_WEB_PORT=8080
docker compose -f ops/compose.prod.yml up -d --build --wait
docker compose -f ops/compose.prod.yml ps
```

默认线索会保存到 `xinghexunjing-leads-data` 卷内的 `/data/leads.jsonl`。查看最近线索：

```bash
docker compose -f ops/compose.prod.yml exec xinghexunjing-leads sh -c 'tail -n 20 /data/leads.jsonl'
```

如果要改用外部 CRM、企业微信、飞书或后端 API：

```bash
export XINGHE_LEAD_WEBHOOK_URL="https://your-crm.example.com/api/xinghe/leads"
docker compose -f ops/compose.prod.yml up -d --build --wait
```

本地打开：

```text
http://localhost:8080/
```

容器启动时会通过 `ops/docker-entrypoint.d/40-render-runtime-config.sh` 生成 `/usr/share/nginx/html/runtime-config.js`。
`XINGHE_LEAD_WEBHOOK_URL` 只能为空、使用 `https://`，或使用同域 `/` 路径，避免公网 HTTPS 页面向 HTTP 线索接口提交导致混合内容失败。

## 生成可上传发布包

独立域名或站点根目录部署使用默认构建：

```bash
npm run release:package
```

Gitee Pages 或其它子路径部署，例如 `https://xinghetech.gitee.io/xinghexunjing/`，必须在打包时设置路径前缀：

```bash
VITE_BASE_PATH=/xinghexunjing/ npm run release:package
```

该变量会影响 `index.html` 内 JS、CSS、`runtime-config.js`、`favicon.svg` 和 `site.webmanifest` 的访问路径。独立域名部署不要设置该变量。

也可以生成 Gitee Pages 专用目录：

```bash
VITE_BASE_PATH=/xinghexunjing/ \
XINGHE_LEAD_WEBHOOK_URL="https://your-crm.example.com/api/xinghe/leads" \
node scripts/prepare-gitee-pages.mjs
```

输出目录为 `tmp/gitee-pages/`。该目录会额外写入：

- `404.html`：静态托管回退页。
- `.spa`：Gitee Pages SPA 回退标记。
- `.nojekyll`：避免静态托管对下划线资源做额外处理。
- `README.md`：记录 commit、basePath 和目标访问地址。

发布时把 `tmp/gitee-pages/` 的内容作为 Gitee Pages 源分支根目录；可以直接执行：

```bash
node scripts/publish-gitee-pages.mjs
```

默认会把 `tmp/gitee-pages/` 内容强制发布到远端 `gh-pages` 分支。也可以通过 `GITEE_PAGES_BRANCH`、`GITEE_PAGES_REMOTE_URL` 和 `GITEE_PAGES_COMMIT_MESSAGE` 覆盖目标分支、远端和提交信息。Gitee Pages 服务本身仍需要在 Gitee 项目后台启用或更新，并把源分支配置为实际发布分支。

发布分支推送后，运行：

```bash
node scripts/verify-gitee-pages-publication.mjs
```

该命令会区分三类问题：

- `deploy-branch`：远端是否存在 `gh-pages` 发布分支。
- `public-repository`：Gitee 仓库是否能被公网访问。
- `gitee-pages-url`：Gitee Pages 地址是否返回官网内容。

如果 `public-repository` 返回 401、403 或 404，先处理仓库公开访问或私有仓库 Pages 服务；如果 `gitee-pages-url` 返回 404，进入 Gitee 项目后台启用或更新 Gitee Pages，并确认源分支是 `gh-pages`。

`npm run release:package` 会先执行 `release:check`，再在 `tmp/releases/` 下生成：

```text
xinghexunjing-web-YYYYMMDD-HHMMSS-<commit>.tar.gz
xinghexunjing-web-YYYYMMDD-HHMMSS-<commit>.manifest.json
```

压缩包结构：

```text
site/
  index.html
  runtime-config.js
  healthz.json
  sitemap.xml
  assets/
ops/
  nginx.conf
  nginx.compose.conf
  nginx.https.conf.template
  compose.prod.yml
  deploy-static-release.sh
  docker-entrypoint.d/
    40-render-runtime-config.sh
services/
  lead-capture-server.mjs
docs/
  官网运营配置说明.md
release-manifest.json
```

上传到服务器后，可以把 `site/` 目录作为 Nginx 站点根目录，也可以使用包内 `ops/deploy-static-release.sh` 执行版本化发布。

服务器发布示例：

```bash
export SITE_ROOT=/www/wwwroot/xinghexunjing
export XINGHE_LEAD_WEBHOOK_URL="https://your-crm.example.com/api/xinghe/leads"
./ops/deploy-static-release.sh /tmp/xinghexunjing-web-YYYYMMDD-HHMMSS-<commit>.tar.gz
```

脚本会把包内 `site/` 发布到 `SITE_ROOT/releases/<release-id>/`，重写 `runtime-config.js`，切换 `SITE_ROOT/current` 软链，并在 `nginx -t` 通过后 reload Nginx。HTTPS 站点可基于包内 `ops/nginx.https.conf.template` 替换 `{{DOMAIN}}`、`{{SITE_ROOT}}`、`{{SSL_CERTIFICATE}}`、`{{SSL_CERTIFICATE_KEY}}` 和 `{{YUDAO_APP_API_UPSTREAM}}`。

如果同域承载星河寻境 APP API，`{{YUDAO_APP_API_UPSTREAM}}` 必须指向 Yudao Server，例如：

```nginx
set $xunjing_yudao_app_api_upstream http://127.0.0.1:48080;
```

部署后用以下命令确认 `/app-api/` 没有被静态站 Nginx 吃掉：

```bash
curl -i -H 'tenant-id: 1' \
  'https://your-domain.example/app-api/xunjing/resource/package?packageCode=KASHGAR-MAP-001'
```

## 健康检查

部署后可用以下路径做负载均衡或监控探针：

```text
/healthz.json
```

预期返回：

```json
{
  "status": "ok",
  "service": "xinghexunjing-web",
  "version": "0.1.0"
}
```

## 部署后验证

公网域名、Nginx、CDN 或对象存储静态站点切换完成后，使用真实访问地址运行：

```bash
npm run deploy:verify -- https://your-domain.example/
npm run deploy:verify -- https://xinghetech.gitee.io/xinghexunjing/
```

该命令会验证：

- 首页返回 200 且包含星河寻境品牌文案。
- 首页包含生产安全响应头：`Content-Security-Policy`、`Referrer-Policy`、`X-Content-Type-Options`、`X-Frame-Options`、`Permissions-Policy`。
- `/healthz.json` 返回 `xinghexunjing-web` 且状态为 `ok`。
- `/runtime-config.js` 可访问，并带有 `Cache-Control: no-store`。
- `/sitemap.xml` 至少包含首页和产品能力锚点。
- `/social-share.svg` 可访问，并保持 1200x630 社交分享图比例。
- 如使用 Compose 默认线索服务，`/api/leads/healthz` 返回 `xinghexunjing-leads ok`。

正式对外运营前，还要运行公网 go-live 门禁：

```bash
node scripts/verify-go-live.mjs https://your-domain.example/
```

该命令会复用 `deploy:verify` 的所有检查，并额外要求：

- 访问地址必须是 `https://`。
- 首页响应头包含 `Strict-Transport-Security`，且 `max-age` 不低于 15552000。
- `runtime-config.js` 必须配置真实的 HTTPS `leadWebhookUrl`，或配置同域 `/api/leads` 且该服务健康；不能为空，也不能使用 `your-crm.example.com` 等示例域名。
