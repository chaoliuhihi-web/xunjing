# 星河寻境业务平台部署说明

更新时间：2026-06-28

## 1. 适用范围

本文档适用于星河寻境 / 图秀中华业务平台部署，包括：

- Yudao Server
- Yudao Admin Vue3
- 小程序 `/app-api`
- 运营后台 `/admin-api`
- MySQL
- Redis
- Object Storage / MinIO
- Qdrant
- AI Key 与模型配置
- 日志、备份、回滚

静态官网部署不使用本文档，见 `docs/02_开发规划/部署上线说明.md`。

## 2. 部署原则

- 星河寻境使用独立 Yudao 副本，不连接 XingheAI2026V2 的运行后台。
- 生产环境、预发环境、开发环境使用独立数据库和独立 Qdrant collection。
- 真实密钥只进入服务器环境变量、密钥管理或 Yudao AI 管理，不进入 Git。
- 所有业务 API 从 Yudao 提供，不新增长期独立 `services/api`。

## 3. 服务拓扑

## 6. 本地依赖栈

本地或预发联调可先启动独立依赖栈，不连接 XingheAI2026V2 的 MySQL、Redis、Qdrant 或对象存储：

```bash
cp ops/xunjing-platform.env.example ops/xunjing-platform.env
npm run xunjing:platform:deps:config
npm run xunjing:platform:deps:up
```

配置好未提交的 `ops/xunjing-platform.env` 后，可把默认 Chat 模型安全写入 Yudao AI 管理表：

```bash
npm run xunjing:ai:bootstrap -- --env-file ops/xunjing-platform.env
npm run xunjing:ai:bootstrap -- --env-file /secure/path/production.env --evidence-file qa/xicheng-yudao-ai-bootstrap-evidence.json
```

该脚本读取 `QWEN_API_KEY`、`QWEN_BASE_URL`、`QWEN_MODEL` 和 MySQL 连接变量，向 `ai_api_key`、`ai_model` 写入或更新默认 Chat 模型，并在生成证据前调用一次 OpenAI-compatible `/chat/completions` 做真实 provider smoke。它通过 `MYSQL_PWD` 和 stdin 传参，不会把真实 key 写入 SQL、Markdown 或 Git，也不会在日志里打印真实 key；provider smoke evidence 只记录 host、endpoint、model、HTTP 状态、响应长度和耗时，不保存回答正文或 key。脚本优先使用本机 `mysql` 客户端；如果本机未安装但 Docker 可用，会自动使用一次性 `mysql:8.4` 客户端容器连接数据库。默认平台为 `TongYi`，可用 `YUDAO_AI_PLATFORM`、`YUDAO_AI_API_KEY_NAME`、`YUDAO_AI_CHAT_MODEL_NAME` 覆盖名称。生产/预发发布前必须用 `--evidence-file qa/xicheng-yudao-ai-bootstrap-evidence.json` 留存 `YUDAO_AI_MODEL_BOOTSTRAPPED` 证据，release gate 会通过 `--ai-bootstrap-evidence` 或 `YUDAO_AI_BOOTSTRAP_EVIDENCE` 校验 `tenantId`、`platform`、`model`、`checkedAt` 和 `ai-provider-smoke`，且证据不得包含数据库密码或 AI key。

OCR/视觉服务不能只检查 `XUNJING_VISION_API_URL`、`XUNJING_VISION_API_KEY` 和 `XUNJING_VISION_MODEL` 是否存在。预发/生产发布前必须用真实可外网访问的西城测试图跑一次 OpenAI-compatible multimodal `/chat/completions` smoke，并保存 `XICHENG_VISION_OCR_SMOKE_READY` 证据：

```bash
npm run xunjing:vision:smoke -- \
  --env-file /secure/path/production.env \
  --image-url https://your-cdn.example.com/xicheng/smoke/baitasi-test-card.jpg \
  --evidence-file qa/xicheng-vision-ocr-smoke-evidence.json
```

该证据只记录 provider host、endpoint path、model、HTTP 状态、响应长度、labels、sample image 引用和耗时，不保存 API key、请求体或模型完整回答；`--evidence-file` 只能写入 `qa/`、`tmp/` 或 `workbench/`。release gate 通过 `--vision-ocr-evidence qa/xicheng-vision-ocr-smoke-evidence.json` 或 `XICHENG_VISION_OCR_EVIDENCE` 校验 `vision-provider-request`、`vision-provider-smoke` 和 `secret-redaction`，没有真实 smoke evidence 时仍输出 `NOT_READY`。

对象存储也不能只检查 `OSS_ENDPOINT`、`OSS_BUCKET`、`OSS_PREFIX`、`OSS_ACCESS_KEY` 和 `OSS_SECRET_KEY` 是否存在。预发/生产发布前必须对真实 bucket 执行一次 S3-compatible 写入、读回和删除 smoke，并保存 `XICHENG_OBJECT_STORAGE_SMOKE_READY` 证据：

```bash
npm run xunjing:storage:smoke -- \
  --env-file /secure/path/production.env \
  --evidence-file qa/xicheng-object-storage-smoke-evidence.json
```

该脚本默认使用 path-style S3 请求；如生产 Yudao 文件配置使用虚拟主机风格，设置 `OSS_PATH_STYLE=false`，如 provider 不能从 endpoint 推断区域，设置 `OSS_REGION`。证据只记录 endpoint host、request host、bucket、prefix、object key SHA-256、region、pathStyle、PUT/GET/DELETE HTTP 状态、读回是否匹配和删除状态，不保存 `OSS_ACCESS_KEY`、`OSS_SECRET_KEY` 或上传正文。release gate 通过 `--object-storage-evidence qa/xicheng-object-storage-smoke-evidence.json` 或 `XICHENG_OBJECT_STORAGE_EVIDENCE` 校验 `object-storage-request`、`object-storage-write`、`object-storage-read`、`object-storage-delete` 和 `secret-redaction`。本地 MinIO 联调可加 `--allow-local`，但本地 HTTP evidence 不得作为预发/生产上线证据。

本地 compose 依赖默认宿主机端口为 MySQL `33306`、Redis `36379`、Qdrant `36333/36334`、MinIO `39000/39001`，避免抢占本机已有 MySQL/Redis。Redis 本地默认密码为 `xunjing_local_redis_password`，应用侧 `.env.local` 需要同步设置 `XUNJING_TENANT_ID=1`、`MYSQL_PORT=33306`、`REDIS_PORT=36379`、`REDIS_PASSWORD=xunjing_local_redis_password`、`QDRANT_URL=http://127.0.0.1:36333`、`OSS_ENDPOINT=http://127.0.0.1:39000`。如本机端口仍冲突，在 `ops/xunjing-platform.env` 中调整 `XUNJING_*_PORT`。

## 7. Nginx 配置要点

```nginx
server {
    listen 443 ssl http2;
    server_name platform.example.com;

    root /opt/xinghe-xunjing/admin/current;
    index index.html;

    location /admin/ {
        try_files $uri $uri/ /index.html;
    }

    location /admin-api/ {
        proxy_pass http://127.0.0.1:48080/admin-api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /app-api/ {
        proxy_pass http://127.0.0.1:48080/app-api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

`review-checklist` 用于快速查看每个点位还缺哪些审核字段；`review-workbook` 把 `aliases`、`ocrKeywords`、`photoLabels`、`fieldEvidenceRefs`、`recommendedQuestions` 等数组字段用 `|` 分隔，供审核人补齐真实资料后再导入 JSON manifest；`review-packet` 记录本批草稿、固定 evidence 文件名、`npm run xunjing:xicheng:poi:tasks:export` 任务导出命令和下一步门禁命令，方便交给人工审核或其它 AI 继续，但它也不是 production evidence。`review:pack` 自动生成的 review pack evidence、workbook evidence 和任务 CSV 只用于暴露待补项，仍不能替代填完真实资料后的 workbook gate、manifest gate 和 seed gate。审核人填完字段表后，先用 importer 生成 `workbench/xicheng-production-pois.json`，再运行 manifest gate；importer 只转换字段结构，不证明资料真实或已授权，但会在 manifest 的 `sourceWorkbook.workbookSha256` 中记录本次输入 workbook 的 SHA-256，方便后续审核追溯到具体 CSV 内容：

只有 workbook gate 输出 `status=XICHENG_POI_REVIEW_WORKBOOK_READY`，且 `summary.workbookRows>=80`、`summary.placeholderCount=0`，才能进入 importer。该 gate 会 fail-closed 检查 80 条以上西城 P0 点位、8 类以上 POI 分类、稳定 `xicheng-*` 编码、GCJ02 坐标、HTTPS 来源、来源授权证据、现场证据引用、触发烟测、内容/坐标/授权审核状态，以及是否仍含 `TODO`、`REVIEW_REQUIRED` 等占位值；输出的 `qa/xicheng-poi-review-workbook-evidence.json` 只证明 CSV 填报完整度，不替代 manifest gate 和 seed gate。未通过时，`summary.workbookReadyPoiCount` 和 `summary.workbookPendingPoiCount` 会给出行级已完成/待补 POI 数，`summary.pendingPoiCodes` 会列出待补点位，`summary.pendingPoiTasks` 会按 CSV 行号输出每个待补点位的 `poiIndex`、`workbookRowNumber` 和 `blockerGroups`；`npm run xunjing:xicheng:poi:tasks:export` 可把这些待补项展开为 `workbench/xicheng-poi-review-tasks.csv`，按 `ownerLane` 分给资料、授权、现场核验、内容审核和占位清理负责人，并在 `taskDetail`、`requiredEvidence`、`workbookColumns` 中写明要补的具体证据和字段列。任务导出 stdout 的 `summary.ownerLaneBreakdown` 还会按负责人汇总 `taskCount`、`poiCount`、`poiCodes`、`blockerGroups` 和 `workbookColumns`，方便另一个 AI 先按 lane 认领范围，再逐行更新 workbook。`summary.blockerBreakdown` 会按 `poi-identity`、`poi-source-license`、`poi-field-evidence`、`poi-content-audit`、`no-placeholder-cells` 等检查项统计 blocker 数，方便发布审核先看整体阻塞分布。

```bash
npm run xunjing:xicheng:poi:manifest:from-workbook -- --workbook workbench/xicheng-production-pois.review-workbook.csv --output workbench/xicheng-production-pois.json --production-ready --batch-code xicheng-p0-poi-review-YYYYMMDD --data-owner xicheng-cultural-tourism-review-team --source-compiled-by xicheng-source-compiler --source-compiled-at YYYY-MM-DD --reviewed-by xicheng-production-reviewer --reviewed-at YYYY-MM-DD --evidence-package-ref oss://xunjing-review/xicheng/review-batches/xicheng-p0-poi-review-YYYYMMDD.zip
```

真实 manifest 应放在 `workbench/` 或受控交付目录，字段至少包含 `regionCode=beijing-xicheng`、`packageCode=XICHENG-MAP-001`、`productionReady=true`、根级 `reviewBatch`、80 条以上 POI、别名、GCJ02 坐标、OCR/照片触发、HTTPS 来源、授权状态、来源授权证据引用、内容摘要、推荐问题、审核记录和现场证据：


根级 `reviewBatch` 必须包含 `batchCode`、`dataOwner`、`sourceCompiledBy`、`sourceCompiledAt`、`reviewedBy`、`reviewedAt` 和非本地 `evidencePackageRef`，用于把整批 80+ POI 绑定到同一个资料整理人、审核人和证据包。`evidencePackageRef` 只允许使用 `https://`、`oss://`、`cos://` 或 `s3://` 这类可审计引用，不允许 `file://`、base64 或本机路径。

manifest gate 通过后，evidence summary 会记录 `summary.manifestSha256`；若 manifest 来自 workbook importer，还会记录 `summary.sourceWorkbookSha256` 和 `summary.sourceWorkbookFile`，用于追溯审核人填写的 CSV 输入。

```bash
npm run xunjing:xicheng:poi:seed:generate -- \

  --output workbench/xicheng-poi-production-seed.sql \
```

生成的 `xicheng-poi-production-seed.sql` 会写入 `xunjing_poi`、POI 级 `xunjing_knowledge_document`、`xunjing_map_point` 和生产 POI 状态报告，并把每个 POI 的来源授权证据和现场证据状态写入审计 JSON；生成前会再次调用 manifest gate，确保未通过审核的数据不会生成 SQL。`xicheng-poi-production-seed-generation-evidence.json` 会记录本次使用的 manifest 文件、manifest SHA-256、`reviewBatchCode`、`reviewBatchEvidencePackageRef`、生成 SQL 文件和 SQL SHA-256，方便审计生成链路；它只能写入 `qa/`、`tmp/` 或 `workbench/`，不能替代 seed SQL gate。生成 SQL 会先执行 `xunjing_assert_xicheng_production_seed_ready` 前置检查：若当前租户缺少 `XICHENG-MAP-001` 资源包或其项目/学校引用，会用 `SIGNAL SQLSTATE '45000'` 直接中止，避免把生产 POI 写入错误上下文。人工审查并准备纳入正式迁移前，必须再运行 seed SQL gate，确认 SQL 仍包含 80+ 个 `xicheng-*` POI、全量审核状态、生产 metrics、批次审核编码、来源授权证据、现场证据状态、POI 级来源文档和该前置检查：

```bash
npm run xunjing:xicheng:poi:seed:verify -- --sql workbench/xicheng-poi-production-seed.sql --evidence-file qa/xicheng-poi-production-seed-evidence.json
```

只有输出 `status=PRODUCTION_POI_SEED_READY`，该 SQL 才能进入预发迁移审查；输出 `NOT_READY` 时以 `blockers` 为准修正 manifest 或重新生成 SQL。该 gate 会逐个校验 `licenseEvidenceRef` 和 `fieldEvidenceRefs` 是非本地 HTTPS、`oss://`、`cos://` 或 `s3://` 引用，普通文本、本地文件路径和 base64 图片不能作为生产来源授权或现场证据。该 gate 不能替代人工 SQL review 和 staging 导入验证。

production seed SQL 导入预发或生产库时，必须使用受控命令先校验 seed evidence 与 SQL hash，再通过 MySQL stdin 写入目标 Yudao 库，并立即查询运行库生成 runtime seed evidence。默认 `npm run xunjing:yudao:runtime-seed:verify` 只证明本地候选 seed 已落库，会在 `summary.productionReady=false` 时仍输出 `YUDAO_XICHENG_LOCAL_SEED_READY`；预发/生产必须通过 `npm run xunjing:yudao:production-seed:apply` 生成 `YUDAO_XICHENG_PRODUCTION_SEED_READY` 运行库证据。该命令没有 `--confirm-apply-xicheng-production-seed` 时会 fail-closed；seed evidence 不是 `PRODUCTION_POI_SEED_READY`、SQL SHA-256 不匹配、POI 审核/授权/现场证据缺失，或目标运行库仍有 `REVIEW_REQUIRED` 坐标/授权，都会失败：

```bash
npm run xunjing:yudao:production-seed:apply -- \
  --env-file /secure/path/production.env \
  --seed-sql workbench/xicheng-poi-production-seed.sql \
  --seed-evidence qa/xicheng-poi-production-seed-evidence.json \
  --runtime-evidence-file qa/xicheng-yudao-runtime-seed-production-evidence.json \
  --apply-evidence-file qa/xicheng-yudao-production-seed-apply-evidence.json \
  --confirm-apply-xicheng-production-seed
```

只有 apply evidence 输出 `status=YUDAO_XICHENG_PRODUCTION_SEED_APPLIED`，且 runtime evidence 输出 `status=YUDAO_XICHENG_PRODUCTION_SEED_READY`，才能把“西城数据已真实填充到后台运行库”作为发布证据。两份 evidence 只记录库名、租户、POI/知识/地图点数量、seed SQL hash、`localCandidateReady`、`productionReady` 和 `productionBlockers`，不会输出 MySQL 密码或 provider key。release gate 必须通过 `--runtime-seed-evidence` 或 `XICHENG_RUNTIME_SEED_EVIDENCE` 读取这份运行库证据；缺失、过旧、仍为 local-candidate、`productionReady=false` 或仍有 `REVIEW_REQUIRED` 都会 fail-closed。

`--evidence-file` 只允许指向 `qa/`、`tmp/` 或 `workbench/` 下的文件；JSON 证据包只包含检查项、摘要和阻塞项，不包含 env 值或密钥。release evidence 的 `summary.gitCommit`、`summary.gitBranch`、`summary.expectedGitBranch` 和 对预发演练显式覆盖。`summary.appApiBaseUrl` 会记录本次 release gate 使用的真实 APP API 后端域名；`summary.yudaoBaselineSqlFile` 和 `summary.yudaoBaselineSqlSha256` 会记录本次验收使用的完整 Yudao baseline SQL 及其 SHA-256；`summary.yudaoServerJarFile`、`summary.yudaoServerJarSizeBytes` 和 `summary.yudaoServerJarSha256` 会记录本次可部署 Yudao Server jar；`summary.aiBootstrapEvidenceFile`、`summary.aiBootstrapModel` 和 `summary.aiBootstrapCheckedAt` 会记录默认 Chat 模型落库证据；`summary.visionOcrEvidenceFile`、`summary.visionOcrModel` 和 `summary.visionOcrProviderSmokeHost` 会记录 OCR/视觉 provider smoke 证据；`summary.objectStorageEvidenceFile`、`summary.objectStorageBucket` 和 `summary.objectStorageProviderSmokeHost` 会记录对象存储写读删 smoke 证据；`summary.runtimeSeedEvidenceFile`、`summary.runtimeSeedProductionReady`、`summary.runtimeSeedPoiTotal`、`summary.runtimeSeedKnowledgeDocuments` 和 `summary.runtimeSeedMapPoints` 会记录目标 Yudao 运行库真实 seed 复验证据。最终发布证据包会重新读取 baseline 和 jar 复算 hash，并要求 release evidence 包含 `release-source-revision`、`yudao-ai-model-bootstrap`、`vision-ocr-service`、`object-storage`、`xicheng-runtime-seed-evidence`、`summary.gitCommit` 与 `summary.appApiBaseUrl`，不能只口头说明 baseline 已导入、jar 已构建、AI 默认模型已配置、OCR key 已配置、上传凭证已配置、代码已同步、西城 seed SQL 已生成或 APP 联调用的是同一域名。`--vision-ocr-evidence` 必须来自 `npm run xunjing:vision:smoke` 且状态为 `XICHENG_VISION_OCR_SMOKE_READY`，并证明真实 provider xunjing:xicheng:poi:workbook:gate` 且状态为 `XICHENG_POI_REVIEW_WORKBOOK_READY`，并证明 `summary.workbookReadyPoiCount>=80`、`summary.workbookPendingPoiCount=0`、`summary.pendingPoiCodes=[]`、`summary.pendingPoiTasks=[]`；`--poi-manifest-evidence` 必须来自 `npm run xunjing:xicheng:poi:manifest:gate` 且状态为 `PRODUCTION_POI_MANIFEST_READY`；`--poi-seed-evidence` 必须来自 `npm run xunjing:xicheng:poi:seed:verify` 且状态为 `PRODUCTION_POI_SEED_READY`；`--runtime-seed-evidence` 必须来自 `npm run xunjing:yudao:runtime-seed:verify -- --mode production` 且状态为 `YUDAO_XICHENG_PRODUCTION_SEED_READY`。release gate 会读取 seed evidence 的 `summary.sqlFile` 做 80+ POI、`productionReady=true`、全量审核状态和 POI 级来源文档检查，再读取 runtime seed evidence 证明同一批数据已经进入目标 Yudao 运行库；不再要求把本地候选 `xunjing-seed-xicheng-p0.sql` 直接改成生产 seed。release evidence 会把 `poiManifestFile`、`poiManifestSha256`、`workbookEvidenceFile`、`sourceWorkbookFile`、`sourceWorkbookSha256`、`poiSeedSqlFile` 和 `poiSeedSqlSha256` 提升到 summary，方便发布审核直接看到本批 POI 的 workbook、manifest、seed SQL 必须能和当前源文件内容匹配；预发演练确需调整时，用 `--max-evidence-age-hours` 或 `XICHENG_MAX_EVIDENCE_AGE_HOURS` 显式声明。只有 release gate 输出 `status=PRODUCTION_READY_CANDIDATE` 才能进入生产发布审核；输出 `NOT_READY` 时必须把 `blockers` 作为 release evidence 附件，不得把本地候选描述为生产上线。若需要把失败项拆给不同负责人处理，可从 release evidence 导出协作任务 CSV；该 CSV 只是修复清单，不是 production evidence：

```bash
  --expected-branch feature/xicheng-p0 \

  --release-evidence qa/xicheng-yudao-release-evidence.json \
  --output workbench/xicheng-yudao-release-blocker-tasks.csv
```

`npm run xunjing:yudao:release:preflight` 会先运行 release gate 并写出 `qa/xicheng-yudao-release-evidence.json`，再自动导出 `workbench/xicheng-yudao-release-blocker-tasks.csv`；命令退出码仍跟 release readiness 一致，`NOT_READY` 时会退出非 0，供 CI 或其它 AI 识别阻塞。`workbench/xicheng-yudao-release-blocker-tasks.csv` 会按 `ownerLane` 拆到 `platform-ops`、`app-ops`、`ai-platform`、`vision-ocr`、`storage-ops`、`yudao-deploy`、`poi-data` 或 `release-manager`，并带 `taskDetail`、`requiredEvidence` 和 `verificationCommand`，用于跟踪 HTTPS 域名、微信、AI/OCR、对象存储、完整 baseline、jar、POI evidence、runtime seed evidence 和 Git 修订等阻塞。任务导出和 preflight stdout 的 `summary.ownerLaneBreakdown` 会按负责人汇总 `taskCount`、`checkNames` 和 `verificationCommands`，方便另一个 AI 先按 lane 认领完整检查项，再回到 CSV 逐行处理。POI evidence 会进一步拆成 workbook gate、manifest gate 和 seed verify 三类复验命令；runtime seed evidence 任务会指向 `npm run xunjing:yudao:production-seed:apply`，要求把已审核 production seed 写入目标库并立即生成运行库证据。不能只用 `review:pack`、SQL 文件或单独 runtime verify 代替生产 evidence。该 CSV 只说明修复任务和复验命令，不包含密钥值，也不能替代 release evidence。

## 14. 可执行部署门禁

本地或 CI 先跑静态门禁，确认独立 Yudao 文件、星河寻境 SQL、喀什 P0 seed、Admin API client 和后台文档仍然齐全：

```bash
npm run xunjing:platform:verify:static
```

预发环境完成 MySQL、Redis、Qdrant、对象存储和 Yudao Server 部署后，执行环境隔离门禁。脚本只输出检查项和变量名，不回显真实密钥：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env
```

预发域名可访问后，增加 live 联调门禁，检查 `/admin/`、资源包公开接口和公益报告摘要接口：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1
```

如果小程序或 APP 使用同一个线上域名访问 Yudao App API，必须先确认网关层存在同域代理：

```bash
curl -i \
  -H 'tenant-id: 1' \
  'https://your-domain.example/app-api/xunjing/resource/package?packageCode=KASHGAR-MAP-001'
```

预期 HTTP 200，响应体 `code=0`。如果线上域名暂时无法承载 `/app-api/`，APP 构建时必须设置 `VITE_XUNJING_YUDAO_APP_BASE_URL=https://your-yudao-domain.example/`，让前端直连已部署的 Yudao HTTPS 基址。

本地只启动 `yudao-server.jar` 做后台 API smoke 时，管理端静态页面通常由前端 dev server 或 Nginx 单独承载，不在 jar 内。此时可显式跳过 Admin UI 可达性检查，但预发和生产不建议跳过：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file ops/xunjing-platform.env \
  --base-url http://127.0.0.1:48080 \
  --tenant-id 1 \
  --skip-admin-check
```

需要把公开写接口也纳入预发门禁时，显式增加写检查。该检查会调用 `/app-api/xunjing/resource/events` 新增一次普通访问事件，并从公开资源包读取一条公开素材后上报 `MEDIA_USE`，验证 `xunjing_media_usage_log` 写入链路，因此只在 staging 或灰度环境使用：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-write-check
```

默认 live 门禁会检查资源包公开读取、`/app-api/xunjing/scan/resolve` 扫码解析和公益报告摘要，`--include-write-check` 会额外写入访问事件和素材使用事件。线上域名只要 `/app-api/` 仍返回 Nginx 404，该门禁必须失败，不能进入 APP 灰度。

西城 P0 需要额外证明 `/app-api/xunjing/triggers/resolve` 已读取 `xunjing_poi` 数据库 seed，而不是只靠 Java fallback。预发或本地 Yudao API 可访问后，显式增加西城 trigger 检查；该检查会依次用 OCR、定位和图片标签请求 `xicheng-baitasi`、`xicheng-gongwangfu`、`xicheng-planetarium`，要求返回 `confidence >= 0.85`、`requiresUserConfirm=false`、目标 `poiCode` 和来源列表：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-xicheng-trigger-check
```

西城 APP 主链还需要证明二维码、错误反馈和 AI 来源门禁可真实联调。显式增加 `--include-xicheng-app-check` 后，会检查 `QR-XICHENG-MAP-001` 扫码进入 `XICHENG-MAP-001`，向 `/app-api/xunjing/resource/events` 写入一条 `ERROR_FEEDBACK`，并对 `/app-api/xunjing/ai/chat` 执行两条西城 POI smoke：`xicheng-baitasi` 必须带匹配来源 `PASSED`，无匹配来源的 source guard 测试点必须 `BLOCKED` 且无 sources；两条 AI 响应都必须回显 APP 传入的 `packageCode`、`sceneCode`、`regionCode`、`poiCode` 和 `poiName`，否则 readiness 失败：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-xicheng-app-check
```

需要把真实模型链路也纳入预发门禁时，再显式增加 AI 检查。该检查会调用 `/app-api/xunjing/ai/chat`、`/app-api/xunjing/reading/ask`、`/app-api/xunjing/map/explain`、`/app-api/xunjing/globe/explain`，覆盖通用问答、图书伴读、地图讲解和地球仪讲解四条 P0 App API，因此只在 staging 或灰度环境使用：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-xicheng-app-check \
  --include-xicheng-trigger-check \
  --include-write-check \
  --include-ai-check
```

该门禁会阻断复用 XingheAI2026V2、Hermes、XOS 等上游运行资源的配置，例如共享 MySQL database、Qdrant collection 或对象存储路径。正式生产环境必须使用星河寻境独立资源。

Yudao 默认启用多租户过滤。小程序、联调脚本和 live 门禁访问 `/app-api/xunjing/**` 时必须带 HTTP header `tenant-id: ${XUNJING_TENANT_ID}`；喀什 P0 本地 seed 为 `1`，生产环境以实际租户编号为准。

## 15. 上线后检查

必须通过：

- `/admin` 可打开。
- 管理员可登录。
- `/admin-api/system/auth/login` 正常。
- 线上域名或 `VITE_XUNJING_YUDAO_APP_BASE_URL` 指向的域名已经把 `/app-api/` 代理到 Yudao Server，不能返回 Nginx 404。
- 带 `tenant-id` 请求头访问 `/app-api/xunjing/resource/package?packageCode=KASHGAR-MAP-001` 正常。
- `/app-api/xunjing/resource/events` 可用 `packageCode` 或 `sceneCode` 回传扫码、访问、分享或素材使用事件，后台自动归因到公开资源包和二维码。
- `/app-api/xunjing/resource/events` 可上报 `ERROR_FEEDBACK`；带 `packageCode` 时，`sceneCode` 可以是普通 APP 场景码，不必是二维码场景码。
- `/app-api/xunjing/scan/resolve` 可使用 `sceneCode=QR-KASHGAR-MAP-001` 解析资源包并返回前端目标页。
- `/app-api/xunjing/triggers/resolve` 可使用 `packageCode=XICHENG-MAP-001`、`regionCode=beijing-xicheng`、OCR、定位和图片标签识别西城 POI；后台必须按 `packageCode` 限定 `xunjing_poi.package_id`，不能跨同一区域其它资源包返回 POI。响应回显 `packageCode` 并在 `targetPath` 中携带 `packageCode`、`regionCode` 和 `poiCode`，同时自动写入 `TRIGGER_RESOLVE` 识别事件；事件 payload 不得保存 `imageBase64` 或 `imageUrl`。
- `/app-api/xunjing/ai/chat` 可返回 `packageCode`、`sceneCode`、`regionCode`、`poiCode`、`poiName`、`answer`、`sources`、`safetyStatus` 和 `logId`；西城请求带 `poiCode` 或 `poiName` 时，响应必须回显该上下文，`sources` 必须匹配该 POI，不匹配时返回 `safetyStatus=BLOCKED`。
- `/app-api/xunjing/reading/ask`、`/app-api/xunjing/map/explain`、`/app-api/xunjing/globe/explain` 可复用同一来源门禁。
- 带 `tenant-id` 请求头访问 `/app-api/xunjing/public-report/summary?packageCode=KASHGAR-MAP-001` 可返回公益报告摘要。
- Yudao AI 管理中存在可用默认 Chat 模型时，问答链路会调用 `AiModelService` 获取模型并生成回答；未配置或异常时回退为确定性讲解文本，接口不能返回空回答。
- 独立库初始化会创建 `ai_api_key`、`ai_model`、`ai_knowledge`、`ai_knowledge_document`、`ai_knowledge_segment` 等 Yudao AI 管理表；真实模型 key 不写入 SQL，必须通过 Yudao AI 管理端、密钥管理或受控迁移写入。
- 启用 `ai-quota-rules.cacheEnabled=true` 后，同日同资源包、同二维码、同用户 trace、同场景、同问题的第二次问答命中缓存，AI 调用日志 `cacheHit=true`、`tokenCount=0`。
- `/admin-api/xunjing/console/ai-eval-sets/run` 可使用 staging 资源包运行 AI 评测集，并返回每题 `safetyStatus`、`sourceCount`、`logId` 和失败原因；未知/实时答案用例未明确拒答时返回 `UNKNOWN_ANSWER_POLICY_NOT_MET`。
- `/admin-api/xunjing/console/dashboard?projectId=项目ID` 可返回扫码、问答、素材、报告看板数据。
- `/admin-api/xunjing/console/crawler-sources/run` 可把外部爬虫或图影中华采集结果生成知识/媒体待审核项，不直接发布。
- `/admin-api/xunjing/console/import-items/review` 审核通过后可把待审核资料发布为知识文档，审核拒绝时不进入知识库。
- `targetType=MEDIA` 的导入项审核通过后可进入 `mediaAssets`，公开接口只返回已审核、已授权且 `canPublic=true` 的素材。
- Yudao Admin 菜单 `星河寻境工作台` 可打开 `xunjing/console/index`，并能完成资源包编辑、知识审核、图影中华素材审核、素材公开/AI/宣传开关、二维码启停、资料导入批量审核和公益报告生成。
- 资源包可绑定 Yudao AI 知识库 `aiKnowledgeId`，问答链路可调用 `AiKnowledgeSegmentService.searchKnowledgeSegment`。
- 知识库检索返回 sources。
- Qdrant collection 可访问。
- 文件上传和预览可用。
- AI Key 只在 Yudao AI 管理或环境变量中可见，Git 中不可见。

## 15. P0 上线门禁

- 独立 Yudao 可运行。
- `yudao-module-xunjing` 可运行。
- 图书扫码、地图扫码、RAG 问答、数据看板、公益报告样例可闭环。
- 未审核知识不进入问答。
- 未审核素材不公开、不参与 AI。
- 已审核素材也必须满足 `copyrightStatus=AUTHORIZED` 且 `canPublic=true` 才能进入 App 公开资源包详情。
- `canAiUse=false` 的素材不能进入 AI 生成任务；`canPromotionUse=false` 的素材不能进入官网、PPT、海报等对外宣传物料。
- 采集来源运行可生成知识和图影中华图片待审核项；图片审核通过后进入素材库，公开前仍受 `canPublic` 控制。
- AI 评测集通过：调用 `/admin-api/xunjing/console/ai-eval-sets/run` 后 `passed=true`、`failedCount=0`，每道要求来源的题目 `sourceCount>0` 且有 `logId`，未知/实时答案题目必须明确表达资料不足、无法确认或不能直接回答。
- AI 成本配额生效：`PROJECT`、`SCHOOL`、`PACKAGE`、`QRCODE`、`USER` 五类作用域至少各有一条 staging 验证记录，超限返回 `safetyStatus=BLOCKED`。
- AI 缓存生效：同一 staging 问题第二次调用不再触发模型，日志记录 `cacheHit=true`，且不绕过配额检查。
- 西城 trigger 识别事件生效：带 `packageCode=XICHENG-MAP-001` 的 `/app-api/xunjing/triggers/resolve` 请求必须写入 `xunjing_interaction_event.event_type=TRIGGER_RESOLVE`，且 payload 不含 `imageBase64` 或 `imageUrl`。
- 西城 APP 错误反馈事件生效：带 `packageCode=XICHENG-MAP-001` 的 `/app-api/xunjing/resource/events` 请求可写入 `event_type=ERROR_FEEDBACK`，并保留普通 APP `sceneCode` 和反馈 payload。
- 西城 AI 对话来源约束生效：具体 POI 问答不能使用其他 POI 或泛化西城资料冒充来源，无匹配来源时必须返回 `BLOCKED` 并记录 source guard 日志。
- 喀什 P0 种子脚本已写入 `PROJECT`、`SCHOOL`、`PACKAGE`、`QRCODE`、`USER` 五类默认配额和 5 条固定 AI 评测题，可直接作为 staging 验收起点。
