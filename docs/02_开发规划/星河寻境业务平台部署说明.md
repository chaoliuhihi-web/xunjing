# 星河寻境业务平台部署说明

更新时间：2026-06-28

## 1. 适用范围

本文档适用于星河寻境 / 图秀中华业务平台部署，包括：

- Yudao Server
- Yudao Admin Vue3
- 小程序 `/app-api`
- 运营后台 `/admin-api`
- MySQL
- Redis
- Object Storage / MinIO
- Qdrant
- AI Key 与模型配置
- 日志、备份、回滚

静态官网部署不使用本文档，见 `docs/02_开发规划/部署上线说明.md`。

## 2. 部署原则

- 星河寻境使用独立 Yudao 副本，不连接 XingheAI2026V2 的运行后台。
- 生产环境、预发环境、开发环境使用独立数据库和独立 Qdrant collection。
- 真实密钥只进入服务器环境变量、密钥管理或 Yudao AI 管理，不进入 Git。
- 所有业务 API 从 Yudao 提供，不新增长期独立 `services/api`。

## 6. 本地依赖栈

本地或预发联调可先启动独立依赖栈，不连接 XingheAI2026V2 的 MySQL、Redis、Qdrant 或对象存储：

```bash
cp ops/xunjing-platform.env.example ops/xunjing-platform.env
npm run xunjing:platform:deps:config
npm run xunjing:platform:deps:up
```

配置好未提交的 `ops/xunjing-platform.env` 后，可把默认 Chat 模型安全写入 Yudao AI 管理表：

```bash
npm run xunjing:ai:bootstrap -- --env-file ops/xunjing-platform.env
```

该脚本读取 `QWEN_API_KEY`、`QWEN_BASE_URL`、`QWEN_MODEL` 和 MySQL 连接变量，向 `ai_api_key`、`ai_model` 写入或更新默认 Chat 模型。它通过 `MYSQL_PWD` 和 stdin 传参，不会把真实 key 写入 SQL、Markdown 或 Git，也不会在日志里打印真实 key。脚本优先使用本机 `mysql` 客户端；如果本机未安装但 Docker 可用，会自动使用一次性 `mysql:8.4` 客户端容器连接数据库。默认平台为 `TongYi`，可用 `YUDAO_AI_PLATFORM`、`YUDAO_AI_API_KEY_NAME`、`YUDAO_AI_CHAT_MODEL_NAME` 覆盖名称。

依赖栈文件：

| 文件 | 用途 |
| --- | --- |
| `ops/xunjing-platform.compose.yml` | 独立 MySQL、Redis、Qdrant、MinIO |
| `ops/xunjing-platform.env.example` | 本地端口、独立数据库名、collection、bucket 示例 |
| `ops/xunjing-platform.env` | 本机真实依赖配置，不提交 Git |

MySQL 首次创建 volume 时会通过 `ops/mysql-init/xunjing-init.sh` 按顺序导入。脚本会先尝试导入完整 Yudao 基础 SQL；当前仓库未提交完整 `ruoyi-vue-pro.sql` 时，会跳过该可选文件，并导入 `xunjing-yudao-runtime-minimal.sql` 作为本地候选 App API readiness 的最小运行时兜底。该 minimal baseline 只创建 `system_tenant`、`infra_api_access_log`、`infra_api_error_log` 和租户 1，不替代预发/生产完整 Yudao baseline。随后脚本导入 Yudao AI 管理表、星河寻境业务表、喀什 P0 seed 和西城 P0 本地候选 seed，并强制校验 `ai_api_key`、`ai_model`、`ai_knowledge`、minimal Yudao runtime 表、`xunjing_%` 表、喀什 P0 资源包、西城 `XICHENG-MAP-001` 资源包、不少于 24 个西城已审核本地候选 POI、以及不少于 24 条西城 POI 级已审核来源文档是否生成：

```text
/opt/xunjing-sql/01-ruoyi-vue-pro.sql
/opt/xunjing-sql/02-xunjing-yudao-runtime-minimal.sql
/opt/xunjing-sql/03-yudao-ai-module.sql
/opt/xunjing-sql/04-xunjing-module.sql
/opt/xunjing-sql/05-xunjing-seed-kashgar-p0.sql
/opt/xunjing-sql/06-xunjing-seed-xicheng-p0.sql
```

如需重建本地空库，先确认没有需要保留的数据，再执行：

```bash
npm run xunjing:platform:deps:down -- -v
npm run xunjing:platform:deps:up
```

本地 compose 依赖默认宿主机端口为 MySQL `33306`、Redis `36379`、Qdrant `36333/36334`、MinIO `39000/39001`，避免抢占本机已有 MySQL/Redis。Redis 本地默认密码为 `xunjing_local_redis_password`，应用侧 `.env.local` 需要同步设置 `XUNJING_TENANT_ID=1`、`MYSQL_PORT=33306`、`REDIS_PORT=36379`、`REDIS_PASSWORD=xunjing_local_redis_password`、`QDRANT_URL=http://127.0.0.1:36333`、`OSS_ENDPOINT=http://127.0.0.1:39000`。如本机端口仍冲突，在 `ops/xunjing-platform.env` 中调整 `XUNJING_*_PORT`。

2026-06-21 本机已完成一次真实联调：独立依赖栈启动后，MySQL `yudao_xinghe_xunjing` 内有 `17` 张 `xunjing_%` 表和 Yudao AI 管理表，喀什 P0 seed 使用 `tenant_id=1`，AI 评测题为 `5` 条；Yudao Server 使用 `SPRING_PROFILES_ACTIVE=local` 连接独立 MySQL、Redis、Qdrant、MinIO 后可在 `48080` 启动。

2026-06-27 已补西城 P0 本地候选 seed：新增 `xunjing_poi` 表、`XICHENG-MAP-001` 资源包、24 个西城本地候选 POI、3 条西城通用审核知识文档、24 条 POI 级已审核来源文档和西城 AI 固定评测集。该数据用于本地/预发联调，不等同生产上线；生产前仍需补齐 80 个以上已复核 POI、坐标复核、来源授权、OCR/视觉服务、真实 AI Key、HTTPS 域名和上传凭证。同日已补 `/app-api/xunjing/ai/chat` 的西城 POI 来源约束：请求带 `poiCode` 或 `poiName` 时，命中的 `sources` 必须匹配该 POI；只有其他 POI 或泛化资料时返回 `BLOCKED`，不能用不相关来源生成成功回答。`/app-api/xunjing/triggers/resolve` 请求带 `packageCode=XICHENG-MAP-001` 时会自动记录 `TRIGGER_RESOLVE` 事件，payload 只保留识别上下文和统计字段，不写入 `imageBase64` 或 `imageUrl`。`/app-api/xunjing/resource/events` 已支持 `ERROR_FEEDBACK`，带 `packageCode` 时允许 `sceneCode` 使用普通 APP 场景码，用于识别失败、AI 反馈或素材异常上报。静态 readiness 里的 `xicheng-poi-seed-quality` 会逐行校验西城本地候选 POI 的编码唯一性、别名、坐标、OCR/图片触发、来源、摘要、推荐问题和审核状态，后续扩容到 80 个 POI 时必须继续通过该质量线。fresh local stack 已通过带 `--include-xicheng-app-check --include-xicheng-trigger-check` 的 live readiness；预发/生产仍需完整 Yudao baseline 和真实外部资源。

server {

## 8. 数据库迁移

上线前必须执行：

```text
backend/yudao/sql/mysql/ruoyi-vue-pro.sql
backend/yudao/sql/mysql/yudao-ai-module.sql
backend/yudao/sql/mysql/xunjing-module.sql
staging 或演示环境可执行 backend/yudao/sql/mysql/xunjing-seed-kashgar-p0.sql
staging 或西城候选环境可执行 backend/yudao/sql/mysql/xunjing-seed-xicheng-p0.sql
```

喀什 P0 seed 包含 5 条固定 AI 评测题，覆盖新疆/喀什基础讲解、地图边界和路线不确定、民族文化表达、宗教文化场所表达、未知/实时答案拒答。

生产环境不能用 `xunjing-yudao-runtime-minimal.sql` 替代完整 Yudao baseline。生产合并前必须运行西城 Yudao release gate，它会 fail-closed 检查 profile、HTTPS APP API 域名、微信 AppID、真实 AI/OCR/视觉服务、对象存储、Qdrant 向量检索和 Embedding 运行开关、完整 Yudao baseline、80 个以上西城已复核 POI、POI 来源授权状态，以及 manifest/seed SQL 两份 POI evidence；生产环境必须配置 `SPRING_AI_VECTORSTORE_TYPE=qdrant`、`SPRING_AI_MODEL_EMBEDDING` 为真实 embedding provider，并设置 `DASHSCOPE_EMBEDDING_ENABLED=true`。完整 Yudao baseline 默认检查 `backend/yudao/sql/mysql/ruoyi-vue-pro.sql`；如果生产部署使用外部受控 SQL 包，不把大体量 baseline 提交到 Git，可通过 `--yudao-baseline-sql /secure/path/ruoyi-vue-pro.sql` 或 `YUDAO_BASELINE_SQL` 指向实际文件，release gate 会读取该文件并校验 `system_users`、`system_tenant`、`system_menu`、`system_oauth2_client`、`infra_api_access_log` 等基础表片段。manifest evidence 必须包含 `manifest-review-batch`、`regionCode=beijing-xicheng`、`packageCode=XICHENG-MAP-001`、`reviewBatchCode`、`reviewBatchEvidencePackageRef` 和 `poi-field-evidence`，seed evidence 必须包含 `seed-preconditions`、`review-batch-metrics`、`regionCode=beijing-xicheng`、`packageCode=XICHENG-MAP-001`、`reviewBatchCode`、`reviewBatchEvidencePackageRef`、`field-evidence` 和 `source-license-evidence`，并且两份 evidence 必须有有效 `checkedAt`、默认在 24 小时内生成，还必须携带源文件路径和 SHA-256，release gate 会重新读取源文件校验 hash，旧版、过旧或内容不匹配的 evidence 不得绕过 release gate。脚本只输出检查项、变量名和阻塞项，不回显密钥：

```bash
npm run xunjing:yudao:release:gate -- \
  --stage production \
  --env-file /secure/path/production.env \
  --yudao-baseline-sql /secure/path/ruoyi-vue-pro.sql \
  --poi-manifest-evidence qa/xicheng-poi-manifest-evidence.json \
  --poi-seed-evidence qa/xicheng-poi-production-seed-evidence.json
```

真实西城 POI 资料进入 production seed 前，可先生成结构模板，把真实审核资料补齐后再跑 manifest gate。模板默认输出 80 个占位 POI，`productionReady=false`，且 `poiCode`、来源、审核和现场证据均为待补状态；它用于整理字段，不得作为 production evidence，也不会通过 production manifest gate：

```bash
npm run xunjing:xicheng:poi:manifest:template -- --output workbench/xicheng-production-pois.template.json
```

真实 manifest 应放在 `workbench/` 或受控交付目录，字段至少包含 `regionCode=beijing-xicheng`、`packageCode=XICHENG-MAP-001`、`productionReady=true`、根级 `reviewBatch`、80 条以上 POI、别名、GCJ02 坐标、OCR/照片触发、HTTPS 来源、授权状态、来源授权证据引用、内容摘要、推荐问题、审核记录和现场证据：

```bash
npm run xunjing:xicheng:poi:manifest:gate -- --manifest workbench/xicheng-production-pois.json --evidence-file qa/xicheng-poi-manifest-evidence.json
```

根级 `reviewBatch` 必须包含 `batchCode`、`dataOwner`、`sourceCompiledBy`、`sourceCompiledAt`、`reviewedBy`、`reviewedAt` 和非本地 `evidencePackageRef`，用于把整批 80+ POI 绑定到同一个资料整理人、审核人和证据包。`evidencePackageRef` 只允许使用 `https://`、`oss://`、`cos://` 或 `s3://` 这类可审计引用，不允许 `file://`、base64 或本机路径。

每个 POI 的 `source` 必须包含 `licenseStatus=APPROVED`、非本地 `licenseEvidenceRef`、`licenseReviewedBy` 和 `licenseReviewedAt`，用于证明来源授权已经人工复核；每个 POI 的 `fieldEvidence` 必须包含 `photoEvidenceStatus=APPROVED`、`triggerSmokeStatus=PASSED`、至少一个非本地 `https://`、`oss://`、`cos://` 或 `s3://` 现场照片/证据引用，以及 `verifiedBy`、`verifiedAt`；不得把 `file://`、`data:image` 或 `imageBase64` 写入 manifest 或 seed。

只有输出 `status=PRODUCTION_POI_MANIFEST_READY`，且 checks 包含 `manifest-review-batch`，才能把该批 POI 转成或合并进生产 seed；输出 `NOT_READY` 时以 `blockers` 为准继续补来源、坐标、现场证据、内容审核、批次审核或点位数量。通过后生成可审查 SQL，生成物先放 `workbench/` 或 `qa/`，人工审查无误后再纳入正式迁移：

```bash
npm run xunjing:xicheng:poi:seed:generate -- \
  --manifest workbench/xicheng-production-pois.json \
  --output workbench/xicheng-poi-production-seed.sql \
  --evidence-file qa/xicheng-poi-production-seed-generation-evidence.json
```

生成的 `xicheng-poi-production-seed.sql` 会写入 `xunjing_poi`、POI 级 `xunjing_knowledge_document`、`xunjing_map_point` 和生产 POI 状态报告，并把每个 POI 的来源授权证据和现场证据状态写入审计 JSON；生成前会再次调用 manifest gate，确保未通过审核的数据不会生成 SQL。`xicheng-poi-production-seed-generation-evidence.json` 会记录本次使用的 manifest 文件、manifest SHA-256、`reviewBatchCode`、`reviewBatchEvidencePackageRef`、生成 SQL 文件和 SQL SHA-256，方便审计生成链路；它只能写入 `qa/`、`tmp/` 或 `workbench/`，不能替代 seed SQL gate。生成 SQL 会先执行 `xunjing_assert_xicheng_production_seed_ready` 前置检查：若当前租户缺少 `XICHENG-MAP-001` 资源包或其项目/学校引用，会用 `SIGNAL SQLSTATE '45000'` 直接中止，避免把生产 POI 写入错误上下文。人工审查并准备纳入正式迁移前，必须再运行 seed SQL gate，确认 SQL 仍包含 80+ 个 `xicheng-*` POI、全量审核状态、生产 metrics、批次审核编码、来源授权证据、现场证据状态、POI 级来源文档和该前置检查：

```bash
npm run xunjing:xicheng:poi:seed:verify -- --sql workbench/xicheng-poi-production-seed.sql --evidence-file qa/xicheng-poi-production-seed-evidence.json
```

只有输出 `status=PRODUCTION_POI_SEED_READY`，该 SQL 才能进入预发迁移审查；输出 `NOT_READY` 时以 `blockers` 为准修正 manifest 或重新生成 SQL。该 gate 不能替代人工 SQL review 和 staging 导入验证。

预发或生产候选发布前，还需要把西城 APP 主链和 trigger HTTP readiness 输出保存为证据文件。该文件由 `verify-xunjing-platform-readiness.mjs --evidence-file` 受控写入，只允许落在 `qa/`、`tmp/` 或 `workbench/` 下，必须包含 `QR-XICHENG-MAP-001` 扫码、`ERROR_FEEDBACK` 写入、`/app-api/xunjing/ai/chat` 的 `PASSED/BLOCKED` 双路径，以及 `xicheng-baitasi`、`xicheng-gongwangfu`、`xicheng-planetarium` 三个 trigger smoke：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /secure/path/production.env \
  --base-url https://platform.example.com/ \
  --tenant-id 1 \
  --skip-admin-check \
  --include-xicheng-app-check \
  --include-xicheng-trigger-check \
  --evidence-file qa/xicheng-app-readiness-evidence.json
```

该 readiness 输出会带 `artifactType=xunjing-platform-readiness`，在 `checkedAt` 记录生成时间，在 `summary.baseUrl` 记录实际联调地址，并在 `summary.tenantId` 记录本次 live checks 使用的租户。开启 `--include-xicheng-app-check` 或 `--include-xicheng-trigger-check` 时，summary 还必须带 `xichengRegionCode=beijing-xicheng` 和 `xichengPackageCode=XICHENG-MAP-001`，供最终发布包和 POI evidence 做交叉一致性校验。CLI 仍会在 stdout 打印同一份 JSON，便于 CI 采集；若 `--evidence-file` 指向仓库根目录、密钥目录或仓库外路径会直接失败。最终发布证据包会拒绝缺少有效 `checkedAt` 的 evidence、缺少 `xunjing-platform-readiness` 来源标识的 evidence、缺少 `summary.tenantId` 的 evidence、默认超过 24 小时的旧 evidence，也会拒绝本地 HTTP evidence；生产或预发候选必须使用非本地 HTTPS 地址生成，不能把 `http://127.0.0.1:48080`、`localhost` 或其他本地联调结果当成上线证据。若预发演练确需调整证据时效窗口，使用 `--max-evidence-age-hours` 或 `XICHENG_MAX_EVIDENCE_AGE_HOURS` 显式声明。

需要留存发布证据时，把 evidence 写入仓库允许的临时/验收目录，不要写到仓库根目录或密钥目录：

```bash
npm run xunjing:yudao:release:gate -- \
  --stage production \
  --env-file /secure/path/production.env \
  --yudao-baseline-sql /secure/path/ruoyi-vue-pro.sql \
  --poi-manifest-evidence qa/xicheng-poi-manifest-evidence.json \
  --poi-seed-evidence qa/xicheng-poi-production-seed-evidence.json \
  --evidence-file qa/xicheng-yudao-release-evidence.json
```

`--evidence-file` 只允许指向 `qa/`、`tmp/` 或 `workbench/` 下的文件；JSON 证据包只包含检查项、摘要和阻塞项，不包含 env 值或密钥。release evidence 的 `summary.yudaoBaselineSqlFile` 和 `summary.yudaoBaselineSqlSha256` 会记录本次验收使用的完整 Yudao baseline SQL 及其 SHA-256；最终发布证据包会重新读取该文件复算 hash，不能只口头说明 baseline 已导入。`--poi-manifest-evidence` 必须来自 `npm run xunjing:xicheng:poi:manifest:gate` 且状态为 `PRODUCTION_POI_MANIFEST_READY`；`--poi-seed-evidence` 必须来自 `npm run xunjing:xicheng:poi:seed:verify` 且状态为 `PRODUCTION_POI_SEED_READY`。两份 evidence 默认必须在 24 小时内生成，且 `manifestFile`/`manifestSha256`、`sqlFile`/`sqlSha256` 必须能和当前源文件内容匹配；预发演练确需调整时，用 `--max-evidence-age-hours` 或 `XICHENG_MAX_EVIDENCE_AGE_HOURS` 显式声明。只有 release gate 输出 `status=PRODUCTION_READY_CANDIDATE` 才能进入生产发布审核；输出 `NOT_READY` 时必须把 `blockers` 作为 release evidence 附件，不得把本地候选描述为生产上线。

最终把四份证据合成一个可归档的发布证据包：

```bash
npm run xunjing:xicheng:release:evidence:package -- \
  --stage production \
  --release-evidence qa/xicheng-yudao-release-evidence.json \
  --poi-manifest-evidence qa/xicheng-poi-manifest-evidence.json \
  --poi-seed-evidence qa/xicheng-poi-production-seed-evidence.json \
  --app-readiness-evidence qa/xicheng-app-readiness-evidence.json \
  --evidence-file qa/xicheng-release-evidence-package.json
```

只有输出 `status=XICHENG_RELEASE_EVIDENCE_PACKAGE_READY`，才能把该证据包提交给发布审核；该 gate 会 fail-closed 检查 release gate、完整 Yudao baseline SQL hash、两份 POI evidence、APP 主链 live checks、`evidence-consistency` 和明显密钥泄漏。两份 POI evidence 必须来自当前版本门禁：manifest evidence 必须包含 `manifest-review-batch`、`regionCode=beijing-xicheng`、`packageCode=XICHENG-MAP-001`、`reviewBatchCode`、`reviewBatchEvidencePackageRef`、`poi-source-license`、`poi-field-evidence`、`manifestFile` 和 `manifestSha256`，seed evidence 必须包含 `seed-preconditions`、`review-batch-metrics`、`regionCode=beijing-xicheng`、`packageCode=XICHENG-MAP-001`、`reviewBatchCode`、`reviewBatchEvidencePackageRef`、`source-license-evidence`、`field-evidence`、`sqlFile` 和 `sqlSha256`；最终证据包会重新读取 baseline/manifest/seed 源文件复算 SHA-256，并校验 APP readiness、manifest 和 seed 指向同一西城 region/package 与同一 POI 审核批次。旧版缺 baseline hash、批次审核、生产 seed 前置检查、来源授权证据、现场证据校验项、源文件哈希或 hash 不匹配的 evidence 不能进入最终发布包。

迁移门禁：

- 迁移脚本可重复审查。
- 生产执行前先在 staging 验证。
- 执行前备份 MySQL。
- 失败时回滚到上一 release 和上一数据库备份。

## 14. 可执行部署门禁

本地或 CI 先跑静态门禁，确认独立 Yudao 文件、星河寻境 SQL、喀什 P0 seed、Admin API client 和后台文档仍然齐全：

```bash
npm run xunjing:platform:verify:static
```

预发环境完成 MySQL、Redis、Qdrant、对象存储和 Yudao Server 部署后，执行环境隔离门禁。脚本只输出检查项和变量名，不回显真实密钥：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env
```

预发域名可访问后，增加 live 联调门禁，检查 `/admin/`、资源包公开接口和公益报告摘要接口：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1
```

如果小程序或 APP 使用同一个线上域名访问 Yudao App API，必须先确认网关层存在同域代理：

```bash
curl -i \
  -H 'tenant-id: 1' \
  'https://your-domain.example/app-api/xunjing/resource/package?packageCode=KASHGAR-MAP-001'
```

预期 HTTP 200，响应体 `code=0`。如果线上域名暂时无法承载 `/app-api/`，APP 构建时必须设置 `VITE_XUNJING_YUDAO_APP_BASE_URL=https://your-yudao-domain.example/`，让前端直连已部署的 Yudao HTTPS 基址。

本地只启动 `yudao-server.jar` 做后台 API smoke 时，管理端静态页面通常由前端 dev server 或 Nginx 单独承载，不在 jar 内。此时可显式跳过 Admin UI 可达性检查，但预发和生产不建议跳过：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file ops/xunjing-platform.env \
  --base-url http://127.0.0.1:48080 \
  --tenant-id 1 \
  --skip-admin-check
```

需要把公开写接口也纳入预发门禁时，显式增加写检查。该检查会调用 `/app-api/xunjing/resource/events` 新增一次普通访问事件，并从公开资源包读取一条公开素材后上报 `MEDIA_USE`，验证 `xunjing_media_usage_log` 写入链路，因此只在 staging 或灰度环境使用：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-write-check
```

默认 live 门禁会检查资源包公开读取、`/app-api/xunjing/scan/resolve` 扫码解析和公益报告摘要，`--include-write-check` 会额外写入访问事件和素材使用事件。线上域名只要 `/app-api/` 仍返回 Nginx 404，该门禁必须失败，不能进入 APP 灰度。

西城 P0 需要额外证明 `/app-api/xunjing/triggers/resolve` 已读取 `xunjing_poi` 数据库 seed，而不是只靠 Java fallback。预发或本地 Yudao API 可访问后，显式增加西城 trigger 检查；该检查会依次用 OCR、定位和图片标签请求 `xicheng-baitasi`、`xicheng-gongwangfu`、`xicheng-planetarium`，要求返回 `confidence >= 0.85`、`requiresUserConfirm=false`、目标 `poiCode` 和来源列表：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-xicheng-trigger-check
```

西城 APP 主链还需要证明二维码、错误反馈和 AI 来源门禁可真实联调。显式增加 `--include-xicheng-app-check` 后，会检查 `QR-XICHENG-MAP-001` 扫码进入 `XICHENG-MAP-001`，向 `/app-api/xunjing/resource/events` 写入一条 `ERROR_FEEDBACK`，并对 `/app-api/xunjing/ai/chat` 执行两条西城 POI smoke：`xicheng-baitasi` 必须带匹配来源 `PASSED`，无匹配来源的 source guard 测试点必须 `BLOCKED` 且无 sources：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-xicheng-app-check
```

需要把真实模型链路也纳入预发门禁时，再显式增加 AI 检查。该检查会调用 `/app-api/xunjing/ai/chat`、`/app-api/xunjing/reading/ask`、`/app-api/xunjing/map/explain`、`/app-api/xunjing/globe/explain`，覆盖通用问答、图书伴读、地图讲解和地球仪讲解四条 P0 App API，因此只在 staging 或灰度环境使用：

```bash
node scripts/verify-xunjing-platform-readiness.mjs \
  --env-file /opt/xinghe-xunjing/env/staging.env \
  --base-url https://platform-staging.example.com/ \
  --tenant-id 1 \
  --include-xicheng-app-check \
  --include-xicheng-trigger-check \
  --include-write-check \
  --include-ai-check
```

该门禁会阻断复用 XingheAI2026V2、Hermes、XOS 等上游运行资源的配置，例如共享 MySQL database、Qdrant collection 或对象存储路径。正式生产环境必须使用星河寻境独立资源。

Yudao 默认启用多租户过滤。小程序、联调脚本和 live 门禁访问 `/app-api/xunjing/**` 时必须带 HTTP header `tenant-id: ${XUNJING_TENANT_ID}`；喀什 P0 本地 seed 为 `1`，生产环境以实际租户编号为准。

## 15. 上线后检查

必须通过：

- `/admin` 可打开。
- 管理员可登录。
- `/admin-api/system/auth/login` 正常。
- 线上域名或 `VITE_XUNJING_YUDAO_APP_BASE_URL` 指向的域名已经把 `/app-api/` 代理到 Yudao Server，不能返回 Nginx 404。
- 带 `tenant-id` 请求头访问 `/app-api/xunjing/resource/package?packageCode=KASHGAR-MAP-001` 正常。
- `/app-api/xunjing/resource/events` 可用 `packageCode` 或 `sceneCode` 回传扫码、访问、分享或素材使用事件，后台自动归因到公开资源包和二维码。
- `/app-api/xunjing/resource/events` 可上报 `ERROR_FEEDBACK`；带 `packageCode` 时，`sceneCode` 可以是普通 APP 场景码，不必是二维码场景码。
- `/app-api/xunjing/scan/resolve` 可使用 `sceneCode=QR-KASHGAR-MAP-001` 解析资源包并返回前端目标页。
- `/app-api/xunjing/triggers/resolve` 可使用 `packageCode=XICHENG-MAP-001`、`regionCode=beijing-xicheng`、OCR、定位和图片标签识别西城 POI，并自动写入 `TRIGGER_RESOLVE` 识别事件；事件 payload 不得保存 `imageBase64` 或 `imageUrl`。
- `/app-api/xunjing/ai/chat` 可返回 `answer`、`sources`、`safetyStatus` 和 `logId`；西城请求带 `poiCode` 或 `poiName` 时，`sources` 必须匹配该 POI，不匹配时返回 `safetyStatus=BLOCKED`。
- `/app-api/xunjing/reading/ask`、`/app-api/xunjing/map/explain`、`/app-api/xunjing/globe/explain` 可复用同一来源门禁。
- 带 `tenant-id` 请求头访问 `/app-api/xunjing/public-report/summary?packageCode=KASHGAR-MAP-001` 可返回公益报告摘要。
- Yudao AI 管理中存在可用默认 Chat 模型时，问答链路会调用 `AiModelService` 获取模型并生成回答；未配置或异常时回退为确定性讲解文本，接口不能返回空回答。
- 独立库初始化会创建 `ai_api_key`、`ai_model`、`ai_knowledge`、`ai_knowledge_document`、`ai_knowledge_segment` 等 Yudao AI 管理表；真实模型 key 不写入 SQL，必须通过 Yudao AI 管理端、密钥管理或受控迁移写入。
- 启用 `ai-quota-rules.cacheEnabled=true` 后，同日同资源包、同二维码、同用户 trace、同场景、同问题的第二次问答命中缓存，AI 调用日志 `cacheHit=true`、`tokenCount=0`。
- `/admin-api/xunjing/console/ai-eval-sets/run` 可使用 staging 资源包运行 AI 评测集，并返回每题 `safetyStatus`、`sourceCount`、`logId` 和失败原因；未知/实时答案用例未明确拒答时返回 `UNKNOWN_ANSWER_POLICY_NOT_MET`。
- `/admin-api/xunjing/console/dashboard?projectId=项目ID` 可返回扫码、问答、素材、报告看板数据。
- `/admin-api/xunjing/console/crawler-sources/run` 可把外部爬虫或图影中华采集结果生成知识/媒体待审核项，不直接发布。
- `/admin-api/xunjing/console/import-items/review` 审核通过后可把待审核资料发布为知识文档，审核拒绝时不进入知识库。
- `targetType=MEDIA` 的导入项审核通过后可进入 `mediaAssets`，公开接口只返回已审核、已授权且 `canPublic=true` 的素材。
- Yudao Admin 菜单 `星河寻境工作台` 可打开 `xunjing/console/index`，并能完成资源包编辑、知识审核、图影中华素材审核、素材公开/AI/宣传开关、二维码启停、资料导入批量审核和公益报告生成。
- 资源包可绑定 Yudao AI 知识库 `aiKnowledgeId`，问答链路可调用 `AiKnowledgeSegmentService.searchKnowledgeSegment`。
- 知识库检索返回 sources。
- Qdrant collection 可访问。
- 文件上传和预览可用。
- AI Key 只在 Yudao AI 管理或环境变量中可见，Git 中不可见。

## 15. P0 上线门禁

- 独立 Yudao 可运行。
- `yudao-module-xunjing` 可运行。
- 图书扫码、地图扫码、RAG 问答、数据看板、公益报告样例可闭环。
- 未审核知识不进入问答。
- 未审核素材不公开、不参与 AI。
- 已审核素材也必须满足 `copyrightStatus=AUTHORIZED` 且 `canPublic=true` 才能进入 App 公开资源包详情。
- `canAiUse=false` 的素材不能进入 AI 生成任务；`canPromotionUse=false` 的素材不能进入官网、PPT、海报等对外宣传物料。
- 采集来源运行可生成知识和图影中华图片待审核项；图片审核通过后进入素材库，公开前仍受 `canPublic` 控制。
- AI 评测集通过：调用 `/admin-api/xunjing/console/ai-eval-sets/run` 后 `passed=true`、`failedCount=0`，每道要求来源的题目 `sourceCount>0` 且有 `logId`，未知/实时答案题目必须明确表达资料不足、无法确认或不能直接回答。
- AI 成本配额生效：`PROJECT`、`SCHOOL`、`PACKAGE`、`QRCODE`、`USER` 五类作用域至少各有一条 staging 验证记录，超限返回 `safetyStatus=BLOCKED`。
- AI 缓存生效：同一 staging 问题第二次调用不再触发模型，日志记录 `cacheHit=true`，且不绕过配额检查。
- 西城 trigger 识别事件生效：带 `packageCode=XICHENG-MAP-001` 的 `/app-api/xunjing/triggers/resolve` 请求必须写入 `xunjing_interaction_event.event_type=TRIGGER_RESOLVE`，且 payload 不含 `imageBase64` 或 `imageUrl`。
- 西城 APP 错误反馈事件生效：带 `packageCode=XICHENG-MAP-001` 的 `/app-api/xunjing/resource/events` 请求可写入 `event_type=ERROR_FEEDBACK`，并保留普通 APP `sceneCode` 和反馈 payload。
- 西城 AI 对话来源约束生效：具体 POI 问答不能使用其他 POI 或泛化西城资料冒充来源，无匹配来源时必须返回 `BLOCKED` 并记录 source guard 日志。
- 喀什 P0 种子脚本已写入 `PROJECT`、`SCHOOL`、`PACKAGE`、`QRCODE`、`USER` 五类默认配额和 5 条固定 AI 评测题，可直接作为 staging 验收起点。
- 西城 Yudao release gate 必须通过：`npm run xunjing:yudao:release:gate -- --stage production --env-file /secure/path/production.env --yudao-baseline-sql /secure/path/ruoyi-vue-pro.sql --poi-manifest-evidence qa/xicheng-poi-manifest-evidence.json --poi-seed-evidence qa/xicheng-poi-production-seed-evidence.json` 输出 `PRODUCTION_READY_CANDIDATE`；manifest evidence 必须包含 `manifest-review-batch`，最终证据包还必须通过 `npm run xunjing:xicheng:release:evidence:package` 并输出 `XICHENG_RELEASE_EVIDENCE_PACKAGE_READY`。若任一 gate 输出 `NOT_READY`，以 `blockers` 为准，不得发布。
- 西城协作阶段必须确认 Gitee `origin` 和 GitHub `github` 的 `feature/xicheng-p0` 指向同一提交；生产合并另走审核，不直接推 `master`。
