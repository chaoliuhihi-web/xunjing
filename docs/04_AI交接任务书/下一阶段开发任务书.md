# 下一阶段开发任务书

更新时间：2026-06-21

## 接手目标

只接手后台时，目标不是做“可演示原型”或小程序假数据，而是在独立 Yudao 副本中打透图秀中华新疆首站 P0 后台闭环：

```text
一本书 / 一张地图 / 一个地球仪 -> 扫码 -> 权威讲解 -> AI 问答有来源 -> 使用数据回传 -> 公益报告生成
```

前端小程序由另一位 AI 负责。后台必须提供真实可上线 API、数据表、权限菜单、审核流、AI 配置、评测、成本控制和部署门禁。

## 第一阶段 P0 后台任务

1. 确认 GitHub / Gitee 双远端。当前如果只存在 Gitee `origin`，必须记录事实，不能假设 GitHub 已配置。
2. 保持 `backend/yudao` 为独立 Yudao 副本，不通过 symlink、submodule、直连数据库、共 Redis、共 Qdrant 或共对象存储依赖 XingheAI2026V2。
3. 所有星河寻境业务进入 `yudao-module-xunjing`，不要把文旅业务表塞进 `yudao-module-ai`。
4. 后台 P0 只做图书扫码、地图扫码、地球仪点位基础绑定、RAG 问答、基础素材库、数据看板和公益报告。
5. 图片库 P0 只做素材上传、来源、版权、审核、前端可用开关和调用日志；图片向量、相似搜索、视频摘要、AI 标注后置。
6. 采集 P0 只做“资料导入与采集审核”：PDF/Word、官方 URL、项目方素材包进入待审核，审核后才进知识库或图片库；不做通用爬虫、站点发现、公众号采集和定时爬虫。
7. AI 问答必须带 sources；无来源时不能返回成功态。
8. 建立固定 AI 评测集，覆盖新疆/喀什基础讲解、地图边界、民族文化、宗教文化场所、未知/实时问题拒答。
9. 建立 AI 成本控制：单用户、单二维码、单学校、单项目预算上限，高频问答缓存，模型 fallback，异常告警。
10. 所有真实密钥只放未提交 env、服务器 Secret 或 Yudao AI 管理，不写入源码、Markdown、测试快照或日志。

## 重要边界

- 不要接手小程序前端页面；前端由另一位 AI 负责，但后台 API 必须稳定。
- 不要恢复早期 `services/api` 自建后端路线。
- 不要把官网部署文档当成业务平台部署文档；业务平台部署看 `星河寻境业务平台部署说明.md`。
- 不要把整图热区、小程序截图或 mock JSON 当作上线后端。
- 不要把 `deliverables/`、`workbench/`、`tmp/`、`archive/` 当源码目录。
- 不要提交 `.env`、`.runtime`、真实 API Key、Token、Cookie、Office 锁文件或大型临时生成文件。
- 正式推送代码时需要同步推送到 GitHub 和 Gitee；如果当前只配置了单一 remote，先说明事实，不要假设另一端已推送。

## P0 验收样例

P0 后台验收至少要有：

1. 一本书和 10 个章节可维护。
2. 一张地图和若干点位可维护。
3. 一个地球仪点位可绑定讲解内容或预生成语音。
4. 100 条权威知识可检索，并且 AI 回答返回 sources。
5. 20 个二维码可创建、绑定目标并统计扫码。
6. 一批基础素材有来源、版权、审核、可用开关和调用日志。
7. 5 条固定 AI 评测题可运行。
8. 一份图秀中华新疆首站公益报告样例可生成。
